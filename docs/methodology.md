# Methodology — Gdynia Thermal Map Audit (Phase 1)

## Overview

This document describes the Phase 1 analytical methodology for auditing the public thermal-loss
map platform of Gdynia, Poland ([termalne.obliview.com](https://termalne.obliview.com)).
The goal is to systematically characterise what data are publicly available, how they are served,
and to derive spatial thermal-loss indicators at multiple administrative granularities.

---

## 1. Web Audit (Step A)

### 1.1 Landing-Page Snapshot

The pipeline begins by downloading the static HTML of the landing page and all directly linked
resources.  A `requests`/`httpx` session with polite rate-limiting is used.  The raw HTML is
saved to `data/raw/` with an SHA-256 checksum logged in the source inventory.

**Objectives**
- Identify all `<script>`, `<link>`, and `<img>` tags.
- Extract inline and external JavaScript filenames.
- Record HTTP response metadata (status code, content-type, `Last-Modified`, `ETag`).

### 1.2 Viewer Snapshot

If the platform exposes a separate viewer URL, it is snapshotted in the same manner.  Any
JavaScript map-configuration objects (e.g. Leaflet / OpenLayers initialisation blocks) are
parsed to extract layer definitions, tile URL templates, WMS/WFS endpoints, and CRS information.

### 1.3 JavaScript Config Parsing

JavaScript files are fetched and scanned with regex patterns targeting:

| Pattern | Example |
|---|---|
| WMS `GetMap` URL | `https://…/wms?SERVICE=WMS&…` |
| WMTS tile template | `https://…/{z}/{x}/{y}.png` |
| WFS `GetFeature` URL | `https://…/wfs?SERVICE=WFS&…` |
| JSON config literals | `{"layers": […]}` |

Extracted configs are stored as JSON in `data/interim/js_configs/`.

---

## 2. Network Probing (Step B)

### 2.1 Candidate URL Inventory

All URLs discovered in Step A — plus a set of well-known OGC endpoint suffixes — are assembled
into a candidate list.  The well-known suffixes tested are:

```
/wms, /wmts, /wfs, /ows, /geoserver/wms, /arcgis/rest/services
```

### 2.2 HTTP Probing

Each candidate URL is probed with an HTTP HEAD (falling back to GET) request.  The
`ProbeResult` dataclass records status code, content-type, redirect chain, and latency.

### 2.3 OGC Capabilities Parsing

For URLs returning XML with `WMS_Capabilities`, `WMTS_Capabilities`, or `WFS_Capabilities`
root elements, `owslib` is used to parse layer names, CRS lists, bounding boxes, and
`GetMap`/`GetTile`/`GetFeature` endpoints.  Results populate the Layer Catalog.

---

## 3. Asset Inventory (Step C)

All discovered assets are catalogued in a structured CSV
(`data/interim/source_inventory.csv`) conforming to the `SourceInventoryRecord` schema:

| Field | Description |
|---|---|
| `url` | Canonical URL |
| `inferred_data_type` | `wms`, `wmts`, `wfs`, `geojson`, `geotiff`, `tile`, `config`, `html` |
| `status_code` | HTTP status |
| `content_type` | MIME type |
| `checksum` | SHA-256 of downloaded body |
| `legal_ethics_note` | Manual flag for any access-control concerns |
| `lineage` | How the URL was discovered |

---

## 4. Analytical Database Construction (Step D)

The analytical database is a set of GeoPackage/CSV files in `data/processed/`:

```
data/processed/
  spatial_units.gpkg      ← districts / neighborhoods / grid cells
  layer_catalog.csv       ← all discovered layers
  source_inventory.csv    ← all probed URLs
  indicators_{unit_type}.csv  ← computed indicators per spatial unit
  annotations.csv         ← manual annotation records
```

Spatial units are constructed according to the `spatial_units.preferred` config key
(see [Section 5](#5-spatial-units)).

---

## 5. Spatial Units

Three levels of spatial aggregation are supported, used in order of preference:

| Level | Source | Notes |
|---|---|---|
| Districts | Official Gdynia administrative districts (18 units) | Preferred; stable boundaries |
| Neighborhoods | Sub-district housing estate units | More granular; boundaries less standardised |
| Regular grid | Generated by `spatial_units.grid` at 100 / 250 / 500 m | Fallback when admin boundaries unavailable |

The `build-grid` CLI command generates square-cell grids in EPSG:2180 (Polish national grid)
and clips them to Gdynia's approximate bounding box or to a supplied boundary polygon.

---

## 6. Thermal Indicator Computation (Step E)

Three *scenarios* are defined based on data availability:

### Scenario A — Raster

Applicable when a raster layer (GeoTIFF or WMS/WMTS tiles) of thermal-loss intensity
can be retrieved.  Zonal statistics (mean, median, percentiles, coverage ratio) are computed
per spatial unit using `rasterio` + custom zonal-stats implementation.

### Scenario B — Vector

Applicable when discrete anomaly features (points or polygons) can be retrieved via WFS
or GeoJSON.  Per-unit counts, densities (per ha), and building-level flags are derived.

### Scenario C — Manual Annotation

Always applicable as a fallback.  Human annotators systematically inspect the web viewer
and record observations in the annotation CSV template (see `docs/manual_annotation.md`).
Indicators are aggregated from the annotation records.

The scenario is selected automatically (`mode: auto`) by attempting Scenario A first,
then B, then C.  The `data_source` column in output CSVs records which scenario was used.
